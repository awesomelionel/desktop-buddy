# Privacy Policy

_Last updated: 2026-05-16_

This document describes what data the **Claude Buddy / desktop-buddy** firmware
handles when it runs on your ESP32-S3 device. It is written to match what the
code in this repository actually does. If the code changes, this document
should be updated alongside it.

The firmware runs entirely on your own hardware. There is **no cloud backend,
no account system, and no analytics or crash-reporting service** operated by
the maintainers of this project. The maintainers cannot see your device, your
settings, or anything Claude Desktop sends to it.

## 1. Who is responsible

This is an open-source firmware project hosted at
`github.com/awesomelionel/desktop-buddy`. The "maintainers" referred to below
are the contributors to that repository. The firmware is provided to you as
source code and as pre-built binaries via GitHub Releases.

You — the person flashing and using the device — are the data controller for
any information your device stores or processes.

## 2. Data stored on your device

The following information is persisted to the ESP32's non-volatile storage
(NVS / flash) on your device only. It is not transmitted off the device by the
firmware.

- **Wi-Fi credentials** — SSID and password of the Wi-Fi network you save,
  stored under the `claude-buddy` NVS namespace.
- **Device settings** — device name (defaults to a MAC-derived
  `Claude-XXXX`), Claude live timeout, screen sleep / dim timeouts, backlight
  levels, daily token cap, enabled cards, card order, and the boot card.
- **Bus-stop slots** — up to four 5-digit Singapore bus-stop codes and
  user-chosen labels, if you configure the bus card.

All of the above can be cleared by:

- **Forget Wi-Fi** in the Web UI (or holding the center button for 5 seconds),
  which wipes the Wi-Fi credentials.
- **Reset settings** in the Web UI, which clears the settings namespace.
- **Factory reset** (Web UI arm + on-device 3-second hold), which clears both.

## 3. Data received from Claude Desktop over Bluetooth

When paired, Claude Desktop sends newline-delimited JSON snapshots to the
device over the Nordic UART Service (NUS) BLE link. Each snapshot may
include:

- Task counters (`total`, `running`, `waiting`).
- A short status message (`msg`, truncated to 32 characters).
- Token-usage figures (`tokens_today`, and an optional `usage` object with
  `used`, `remaining`, and `limit`).
- A permission prompt (`prompt` object: an opaque id, a tool name, and a
  short hint).
- Heartbeat snapshots used to mark Claude as live or offline.

**This data is kept in RAM only.** The firmware does not persist any
snapshot content to flash, and does not forward it to any third party. The
latest snapshot is overwritten by the next one and is lost on reboot.

The content of a snapshot is generated by Claude Desktop on your computer.
What ends up inside `msg` or `hint` is determined by Claude Desktop and your
own Claude session — this firmware just renders it.

## 4. Data sent to Claude Desktop over Bluetooth

In response to a permission prompt, the device sends a single JSON message
back over BLE:

```json
{"cmd":"permission","id":"<prompt-id>","decision":"once|deny"}
```

That is the only data the firmware sends to Claude Desktop. No device
identifier, telemetry, location, or settings are transmitted.

## 5. Network requests the device makes

Apart from BLE, the firmware initiates outbound network connections in two
cases:

### 5.1 Firmware updates (GitHub)

When you click **Check for updates** or **Install update**, the device
contacts GitHub over HTTPS:

- `https://api.github.com/repos/awesomelionel/desktop-buddy/releases/latest`
  to read release metadata, then the redirect target on
  `release-assets.githubusercontent.com` to download `firmware.bin`.
- Request headers: `User-Agent: claude-buddy-ota/1`,
  `Accept: application/vnd.github+json`.

The device does **not** include a device identifier, MAC address, or
settings in these requests. GitHub will, however, see the public IP address
your device connects from, which is true of any HTTPS client. GitHub's
handling of that data is governed by GitHub's own privacy policy.

### 5.2 Bus arrival times (optional)

If you enable the bus card and configure a stop code, the device queries:

- `https://api.busaunty.com/api/v1/BusArrival?BusStopCode=<code>`
  with `User-Agent: claude-buddy/1`.

Only the 5-digit stop code you chose is sent. Use of this third-party API is
optional — if you do not configure a bus stop, this request is never made.
This API is operated by a third party and has its own privacy and terms;
the maintainers of this project have no control over it.

There are **no other outbound network calls** in the firmware. No analytics,
no crash reporting, no remote logging.

## 6. The local Web UI

The device hosts an HTTP server on your local network (port 80) and exposes
JSON endpoints (see the README for the full list). This server has **no
authentication**. Anyone on the same Wi-Fi network as the device can:

- Read the saved Wi-Fi SSID, device IP, device name, uptime, and the most
  recent Claude status snapshot via `/api/status` and `/api/settings`.
- Change settings, save new Wi-Fi credentials, scan for Wi-Fi networks,
  trigger an OTA install, reboot, or wipe settings.

Treat the device the same way you would treat any other unauthenticated
device on your LAN. Run it on a trusted network. The maintainers do not
operate or have access to this interface.

## 7. The provisioning access point

On first boot, or after you forget Wi-Fi, the device opens an **unencrypted
Wi-Fi access point** named `claude-buddy-XXXX` (where `XXXX` is derived from
the Wi-Fi MAC) and serves a captive portal at `http://192.168.4.1/`. Anyone
within radio range can join this AP and submit Wi-Fi credentials to it.

Complete provisioning promptly. Once the device joins a Wi-Fi network the AP
is taken down.

## 8. Identifiers

The only stable identifiers the firmware uses are:

- The MAC-derived device name `Claude-XXXX`, advertised over BLE and shown
  on the device and Web UI. You can change it.
- The MAC-derived AP SSID `claude-buddy-XXXX`, visible only while the device
  is in provisioning mode.

The firmware does not generate or transmit any other unique identifier.

## 9. Logs

The firmware writes diagnostic logs to its USB serial port. Those logs are
visible only to whoever is physically attached to the device with a USB
cable running a serial monitor. They are not sent anywhere.

## 10. Children

This firmware is intended for general developer use. It is not directed at
children and does not knowingly collect any data about anyone.

## 11. Changes

If the data the firmware handles changes, this document should be updated in
the same commit. The "Last updated" date at the top reflects the most recent
revision.
